## 2. Data Flow

### Stage 1: EXTRACT
**Input:** NYC Yellow Taxi Parquet file  
**Output:** Pandas DataFrame (500K rows)

\`\`\`python
# Read Parquet with optional sampling
df = pd.read_parquet(path)
df = df.head(sample_size)  # 500K rows
\`\`\`

**Why Parquet?**
- Columnar format (memory efficient)
- Built-in compression (fast reads)
- Preserves data types
- Ideal for large analytical datasets

**Why Sample?**
- Neon PostgreSQL free tier: 512 MB storage limit
- Full file (2.8M rows) would exceed limit
- 500K rows = representative sample of Jan 2023 patterns

---

### Stage 2: TRANSFORM
**Input:** Raw DataFrame  
**Output:** Clean, feature-engineered DataFrame

Key transformations:

1. **Datetime Conversion**
   \`\`\`python
   df['tpep_pickup_datetime'] = pd.to_datetime(...)
   df['tpep_dropoff_datetime'] = pd.to_datetime(...)
   \`\`\`

2. **Feature Engineering**
   \`\`\`python
   df['trip_duration'] = (
       df['tpep_dropoff_datetime'] - df['tpep_pickup_datetime']
   ).dt.total_seconds() / 60  # Convert to minutes
   \`\`\`

3. **Column Standardization**
   \`\`\`python
   # PascalCase → snake_case
   VendorID → vendor_id
   RatecodeID → rate_code_id
   PULocationID → pu_location_id
   DOLocationID → do_location_id
   \`\`\`

4. **Data Validation**
   - Remove trip_duration < 0 or > 1440 min (24 hours)
   - Remove fare_amount < 0
   - Remove passenger_count < 1
   - Log warnings for invalid rows

---

### Stage 3: LOAD
**Input:** Clean DataFrame  
**Output:** Data in PostgreSQL

\`\`\`python
df.to_sql(
    "yellow_taxi_2023",
    engine,
    if_exists="replace",  # or "append" for incremental
    index=False,
    method='multi',
    chunksize=10000  # Batch insert for efficiency
)
\`\`\`

**Database Design:**
- **Connection:** Neon PostgreSQL (cloud, free tier)
- **Table:** \`yellow_taxi_2023\`
- **Rows:** 500,000
- **Schema:** Auto-generated by SQLAlchemy from DataFrame dtypes

**Why PostgreSQL?**
- ACID compliance (data integrity)
- Complex queries (analytics)
- Free tier via Neon (no credit card)
- Production-ready for job portfolios

---

### Stage 4: ANALYTICS
**Input:** PostgreSQL table  
**Output:** Insights, CSV exports, visualizations

Sample queries:

\`\`\`sql
-- Peak hours
SELECT EXTRACT(HOUR FROM tpep_pickup_datetime) AS hour,
       COUNT(*) AS trips
FROM yellow_taxi_2023
GROUP BY hour
ORDER BY trips DESC;

-- Revenue per day
SELECT DATE(tpep_pickup_datetime) AS day,
       SUM(total_amount) AS revenue
FROM yellow_taxi_2023
GROUP BY day
ORDER BY day;
\`\`\`

## 4. Error Handling & Logging

### Logging Levels
- **DEBUG:** Detailed information (variables, intermediate values)
- **INFO:** Progress updates (✓ Extract complete, ✓ Load complete)
- **WARNING:** Non-critical issues (missing values, invalid rows)
- **ERROR:** Critical failures (file not found, DB connection error)

## 5. Configuration Management

**Sensitive data** (.env file) is stored separately:

PARQUET_PATH=/path/to/data.parquet
TABLE_NAME=yellow_taxi_2023
SAMPLE_SIZE=500000
\`\`\`

**Why?**
- Never commit secrets to Git
- Easy to switch between environments (local, Kaggle, production)
- \`.env\` listed in \`.gitignore\`

## 6. Performance Considerations

### Extraction Optimization
- **Parquet format:** ~10x compression vs CSV
- **Sampling:** Reduces memory and processing time
- **Chunked reading:** Not needed for 500K rows, but scalable

### Transform Optimization
- **Vectorized operations:** pandas (not Python loops)
- **Column selection:** Only needed columns processed
- **Batch validation:** Single pass through data

### Load Optimization
- **Batch insert:** \`chunksize=10000\` (insert 10K rows at a time)
- **Index handling:** \`index=False\` (no need for row index)
- **SQLAlchemy ORM:** Automatic type inference from DataFrame

## 7. Security Considerations

**Implemented:**
- Secrets in \`.env\` (not in code)
- \`.gitignore\` prevents credential leaks
- SQL injection protection via SQLAlchemy ORM
- Password never shown in logs

**To Implement in Production:**
- Use environment secrets in CI/CD (GitHub Secrets)
- SSL/TLS for all database connections
- Row-level security (RLS) for multi-tenant scenarios
- Audit logging for compliance
